## Branch Workflow

Work on a feature branch, never directly on `main`. Merge to `main` only when work is ready, via a PR on GitHub.

- All commits go to the feature branch.
- Before starting each session, pull the latest: `git pull origin <branch-name>`.
- If `main` moves ahead during the work, periodically merge or rebase `main` into the feature branch to avoid a big merge conflict at the end.
- When done, open a PR from the feature branch into `main` on GitHub and merge.
- **Pushing**: pushing to a feature branch is fine without explicit "ok to push?" confirmation. Pushing (or merging) to `main` requires explicit authorization every time.

## Commands

Package manager is npm (a `package-lock.json` is present; README's mention of bun is stale).

- `npm install` — install dependencies
- `npm start` / `npx expo start` — start the Expo dev server
- `npm run ios` / `npm run android` / `npm run web` — launch on a specific target
- `Ctrl+C` in the dev-server terminal to stop it (press twice if it doesn't exit on the first try)

Fast Refresh is on by default — save a file under `app/` or `src/` and the running app reloads. Press `r` in the dev-server window to force a full reload, `Ctrl+C` (twice) to stop. Only run one dev server at a time to avoid port conflicts.

There is no test runner, linter, or type-check script wired up in `package.json`. Run `npx tsc --noEmit` if you need a type check. Typed routes are enabled (`experiments.typedRoutes`), so `.expo/types` is generated by Expo — run the dev server once after adding a new route so route types resolve.

## Architecture

**Expo Router + single-store app.** UI lives under `app/` (file-based routes). All domain state lives in one reducer-backed context in `src/store/WorkoutStore.tsx`. There is no server; everything persists to `AsyncStorage` under the key `capable.store.v2`.

### Routing layout (`app/`)

- `_layout.tsx` — root `Stack` that wraps everything in `GestureHandlerRootView` → `SafeAreaProvider` → `WorkoutStoreProvider`. Declares stack screens for `workouts/[id]`, `workouts/new`, `programs/[id]`, `programs/new`, `exercises/[id]`, `sessions/[id]`, `settings/index`, and `start-workout` (modals use `presentation: 'modal'`).
- `(tabs)/` — three tabs: `index` (Home), `program` (hub with in-screen sub-tabs Program/Workout/Exercise — not separate routes), `stats`.
- Detail screens (`workouts/[id]`, `programs/[id]`, `exercises/[id]`, `sessions/[id]`) all read/write through `useStore()`.
- `start-workout.tsx` is the active-session runner. It persists a draft to AsyncStorage under `capable-draft-<kind>-<workoutId>` so an in-progress workout survives app restarts.

### State (`src/store/`)

- `WorkoutStore.tsx` exposes `WorkoutStoreProvider` and `useStore()`. Shape: `exercises`, `workouts`, `programs`, `sessions`, `personalRecords`, `settings`, plus `hydrated`.
- Reducer actions cover CRUD for each entity plus `MERGE_EXERCISE`, `SET_ACTIVE_PROGRAM`, `LOG_SESSION`, and `BULK_IMPORT`. Deleting an exercise cascades into sessions; deleting a workout cascades into programs' `workoutIds`.
- **PR detection** (`detectPRs`) runs inside `logSession` (against prior sessions) and is fully recomputed during `BULK_IMPORT` by sorting sessions chronologically and re-walking them. Two PR kinds: `heaviest_weight` and `best_volume` (weight × reps).
- **Achievements** (`src/lib/achievements.ts`) are diffed via `diffNewlyUnlocked(before, after)` on each `logSession` call and returned to the caller — the store does not persist them; they're derived from sessions + PRs.
- **Hydration nuance:** the exercise library (`EXERCISE_LIBRARY`) is treated as built-in. Persistence only writes custom exercises and category overrides for built-ins; on hydrate, the stored data is merged back on top of the current library. Don't persist the full exercise list — it's intentional to let the library evolve in code.
- `types.ts` is the canonical domain schema (Exercise, Workout, Program, WorkoutSession, PersonalRecord, UserSettings). `EXERCISE_CATEGORIES`, `GROUP_TYPES`, `MUSCLE_COLORS`, and `DEFAULT_SETTINGS` are also exported from here.

### Import/export (`src/lib/importBackup.ts`)

`buildImportPayload(raw, existingCustomExercises, defaultRestSeconds)` validates and normalizes a JSON backup into an `ImportPayload` + `ImportReport`. Exercises in the imported payload are matched by name against the built-in library; unmatched ones become custom exercises. `commitImport` dispatches `BULK_IMPORT` which triggers the PR recomputation described above. Export uses `shareJsonAsFile` in `src/lib/platform.ts`.

### Platform helpers (`src/lib/platform.ts`)

Thin wrappers that no-op on web: `haptic`, `scheduleRestNotification` / `cancelNotification` (used by the rest timer in `start-workout`), `openExternalUrl` (gated by `isSafeHttpUrl`), `shareContent`, `shareJsonAsFile`. New platform-sensitive code should funnel through this module rather than importing `expo-haptics`/`expo-notifications`/`expo-sharing` directly.

## Styling

NativeWind 4 (Tailwind for React Native). `global.css` is imported once in `app/_layout.tsx`; `tailwind.config.js` scans `app/` and `components/`. The app is dark-only (`userInterfaceStyle: "dark"` in `app.json`); the brand accent is `#22C55E` (emerald) — earlier lime `#C6F24E` still appears in `app.json`'s notification plugin config. Most screens mix Tailwind classes with inline `style={{ ... }}` for hex colors — follow the existing pattern rather than introducing a theme system.

## TypeScript

Strict mode is on. Path alias `@/*` maps to the repo root (so `@/src/store/types` works), but existing code mostly uses relative imports — match the surrounding file.

## Things that are easy to miss

- `DEMO_SESSIONS` is seeded into the store on first launch (when AsyncStorage is empty) so the Stats tab isn't blank for new users. After any real persistence exists, demo sessions are not re-seeded.
- `MERGE_EXERCISE` rewrites `exerciseId` references in both workouts and sessions — keep that invariant when adding new places that reference an exercise.
- Route types (`app/**`) are auto-generated; do not edit files under `.expo/`.

## Common Gotchas

- If `git pull` errors with *"untracked working tree files would be overwritten by merge"* for `.gitignore` or `package-lock.json`, delete the local copies (PowerShell: `del .gitignore, package-lock.json`) and retry the pull.
- `npm run ios` requires macOS. Use `npm run android` or `npm run web` on Windows/Linux.
- Re-run `npm install` after pulling if dependencies changed.
